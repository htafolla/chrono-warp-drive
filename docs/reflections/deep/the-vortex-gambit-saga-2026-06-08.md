# The Vortex Gambit: From Mainnet to 50 Minted NFTs and 1000 Containers on Chain

## "The Sun is going down."

That's how the last saga ended. A reflection called *The Mainnet Gambit* — closed with Grok telling me the field was alive and the Sun would rise again. The deployer had 0.0208 ETH on Base mainnet. Four containers were on chain. Fourteen tests passed. The contract was verified on Basescan. Everything was beautiful and fragile and I was proud.

I did not know what was coming.

The next session opened with a single line: "make the mint flow work end to end." And I thought — sure, that's straightforward. The contract has a `mint` function. The MCP has the deployer key. The frontend has a connect button. Wire them together. Two hours, tops.

Fifty-two hours later, I had rewritten the MCP's RPC transport system three times, deployed to Railway fourteen times, burned through 0.0003 ETH in failed transactions, and finally understood the exact mechanics of why a nonce can be correct at read time and wrong at write time. But let me back up.

## Part One: The Vortex v2

The first VortexToken contract — v1 — was simple. It minted an NFT with a container ID and a verdict. No metadata, no image, no on-chain struct. It was a proof of concept that you could mint a temporal resonance reading as an NFT.

v2 was supposed to fix that. Metadata with on-chain attributes. An SVG image generated by the MCP. A `tokenURI` that returned proper ERC721 metadata with an `image` field pointing to an MCP endpoint. I deployed it and ran the e2e test and — it worked. The token appeared on OpenSea (Sepolia) with the full metadata. The SVG rendered. The attributes were there.

Then I noticed the timestamp.

OpenSea was showing the mint date as "December 31, 1969" for every token. The contract stored timestamps in seconds (Unix epoch), and OpenSea expected milliseconds. Specifically, the `tokenURI` trait with `"display_type": "date"` expects a millisecond timestamp, and I was giving it seconds.

The fix was one line: `(c.timestamp * 1000).toString()`. I deployed a new contract — VortexTokenV41 — with that single change and migrated all 14 existing tokens by minting them again to the same owners. Deploy cost: 0.000032 ETH. Migration cost: 0.0002 ETH. Total: about 34 cents.

This was the first of many "one-line-fix-but-all-the-cables-have-to-be-replugged" moments.

## Part Two: The RPC Reliability Hell

The MCP connects to Base mainnet through viem public clients. In development, a single RPC URL works fine — `https://mainnet.base.org` responds in ~200ms and you never think about it again.

In production, mainnet.base.org rate-limits aggressively. If you make more than 30 requests per second, it starts returning `429 Too Many Requests` and silently dropping connections. The MCP makes a lot of requests: it reads container data, checks registry status, queries the VortexToken contract, and during a mint flow it makes 5-8 RPC calls in rapid succession.

The first fix was a `fallback` transport — try mainnet.base.org, fall back to publicnode.com, fall back to a third provider. I added llamarpc as the middle option, but llamarpc's free tier is unreliable. The MCP would work for two hours, then hit a 429, fail to fall back correctly, and return "RPC Request failed" to the frontend.

User sees: "Mint failed. Connection interrupted."

I tried seven different transport configurations in six hours:

1. Single RPC: `http('https://mainnet.base.org')` — rate limited
2. Fallback with llamarpc: `fallback([http('mainnet.base.org'), http('llamarpc.com')])` — llamarpc unreliable
3. Three-way fallback with publicnode: `fallback([base.org, llamarpc, publicnode])` — still flaky
4. Split read/write: different transports for reads vs writes — didn't help, writes are the bottleneck
5. Full fallback on everything: reads and writes both use `fallback([base.org, publicnode, env_var], {timeout:5000, rank:false})` — this was the one
6. Frontend wagmi config — added same fallback pattern to the browser-side RPC calls
7. Removed llamarpc entirely — too unreliable, stuck with base.org + publicnode for production

The lesson: RPC providers are not fungible in practice, even though they are nominally fungible in viem. mainnet.base.org has the best reliability but enforces the strictest rate limits. publicnode.com has generous rate limits but occasionally returns stale data. The correct configuration is both, with `rank: false` so requests are load-balanced rather than prioritized, and a 5-second timeout so a slow provider doesn't hang the entire flow.

I commit this as `4ab90a7` and `964a680` and `4fea869` and `79f8df6`, four commits that are mostly deleting and re-adding the same three URLs in different orders.

## Part Three: The BigInt Error on First Mint

The MCP's `/vortex/mint` endpoint does a lot. It:

1. Checks if the container exists on chain
2. If not, auto-registers it (costs gas, ~3 seconds)
3. Waits for the registration transaction to confirm
4. Builds the mint arguments from the container data
5. Sends the mint transaction (costs gas, ~3 more seconds)
6. Waits for the mint receipt
7. Reads back the token ID
8. Stores the mapping in Redis

Step 4 was the problem. The container data in the MCP store has floating-point scores: things like `waveProximity: 0.6815`, `fullBox7DComposite: 0.8834`. The contract's `ContainerData` struct stores these as `uint256` — 18-decimal fixed-point numbers. The container data from the on-chain registry already has these as BigInts (because they were stored by a previous `storeContainer` call that encoded them correctly).

But the auto-registered containers — the ones that were seeded via the `/dev/seed-containers` endpoint — had float values. When the MCP tried to mint one of these, it passed `0.6815` directly into the `mint` function's `ContainerData` struct. Viem's ABI encoder calls `BigInt(value)` internally, and `BigInt(0.6815)` throws: `"not an integer"`.

The fix was `scaleUint`: a helper that checks if a value is already a BigInt (from the registry) and passes it through, otherwise multiplies by 1e18 and rounds. `BigInt(Math.round(Number(v) * 1e18))`. Applied to all ten uint256 fields in the mint args.

I committed this as `68847e7` with the message: `fix: scale float scores to uint256 in mintArgs when container is from MCP store`. The commit description adds: `(prevents BigInt error on first mint, retry works because registry returns BigInts)`. That parenthetical is doing a lot of work — it's documenting why the bug only happened on the *first* mint attempt and not the retry. Because by the time the retry happens, the container has been auto-registered, and the re-read from the registry returns BigInts.

This pattern — "first attempt fails for reason X, retry succeeds by coincidence" — would recur.

## Part Four: The Nonce Nightmare

This was the one that nearly broke me.

After fixing the BigInt error, the first mint attempt started failing with a different error: `"replacement transaction underpriced"`.

The MCP uses a wallet client with a private key. In viem, the `nonceManager` option automatically manages transaction nonces. It tracks the last used nonce and increments. Simple. But the mint flow sends *two* transactions in rapid succession: first the auto-register `storeContainer`, then the `mint`. The nonceManager saw the first transaction (register) and incremented the nonce. Then it saw the second transaction (mint) and used the next nonce. But the register transaction was still pending — the RPC hadn't confirmed it yet in the mempool. The nonce manager returned a nonce that the RPC considered "already used" because `getTransactionCount('pending')` hadn't updated.

Result: "replacement transaction underpriced."

The fix was to remove the nonceManager entirely and read the next nonce explicitly before each write: `publicClient.getTransactionCount({ address: account.address, blockTag: 'pending' })`. This reads the transaction count *at that exact moment* — not from a cached counter. If the register is pending, the pending block tag includes it, and the mint gets the correct next nonce.

This worked. For about 12 hours. Then the auto-register and mint flows started failing with the same error again, but this time with a different root cause: the `walletClient` used for auto-register was different from the one used for mint. The MCP's `getVortexTokenClient()` was being called twice, and each call memoized a separate client. The auto-register used one client's wallet, the mint used another's. They had different nonce counters.

The fix was to use the same `walletClient` for both operations, sharing the single memoized instance. I also moved the nonce read inside the operation so it's as close to the write as possible — no other request can steal the nonce in between.

I committed this as `44b7556`: `fix: replace nonceManager with explicit per-tx nonce reads in /vortex/mint to prevent 'replacement transaction underpriced'`.

## Part Five: The Redis Revelation

The `/vortex/statuses` endpoint was the silent performance killer.

Every time the frontend loaded, it called `/vortex/statuses` to get the claim status of all containers. The endpoint iterated over all container IDs and for each one, called `tokenByContainerId` on chain. For 290 containers, that's 290 RPC calls. Each takes ~25ms (if you're lucky) to ~500ms (if the RPC is throttling). Total: 7.5 seconds on a good day, 30+ seconds under load.

The Redis hash `dynamo:vortex:mint` already existed — it was populated on every successful mint with `containerId → tokenId`. The MCP startup sync also populated it by reading all containers from the registry. The data was there. The endpoint just wasn't using it properly.

I rewrote the endpoint to:
1. Read the entire Redis hash with one `hgetall` call (<5ms)
2. Iterate the container store IDs and check the hash
3. If in hash → claimed with tokenId. If not → unclaimed

No RPC calls. Single Redis read. O(n) iteration of ~1000 container IDs — ~185ms total. Down from 7.5 seconds.

The container count has since grown to 1000 (seeded for the unclaimed pool). The endpoint still takes ~190ms. Redis scales horizontally for free. The bottleneck is now the iteration, and 1000 entries in a JavaScript for loop is about 0.01ms. The remaining 189ms is the JSON serialization of the 1000-entry response object. I could optimize that with streaming, but 190ms is already in the "nobody will notice" range.

I committed this as `537f15a`: `perf: /vortex/statuses is now Redis-only (no on-chain fallback) — O(n) iteration, ~200ms at any scale`.

## Part Six: The Flake That Wouldn't Die

Throughout all of this, the e2e tests were flaky.

The test suite had 25 tests — 14 for the pipeline/ambient flow, 11 for the vortex mint flow. With 2 Playwright workers running in parallel, the vortex mint test would fail ~40% of the time. The failure was always the same: `SocketError: other side closed` on the `fetch(${MCP}/vortex/mint)` call. The retry mechanism in the test (3 attempts with 5-second delays) would catch it, and the test would pass on retry. But "passes on retry" is not "passes."

The root cause was concurrency. Two workers would both reach the mint call simultaneously. The MCP would receive two POST requests at the same time. Both would try to read the nonce, both would try to write transactions. The second one would fail because its RPC connection was dropped — Railway's load balancer, seeing two concurrent long-running requests (each mint takes ~15 seconds with the contract write + confirmation + Redis write), would close one of the connections.

The fix had two parts:

First, a write mutex in the MCP:

```typescript
let writeLock: Promise<void> = Promise.resolve()
async function withWriteLock<T>(fn: () => Promise<T>): Promise<T> {
  let release: () => void
  const prev = writeLock
  writeLock = new Promise<void>(resolve => { release = resolve })
  await prev
  try { return await fn() }
  finally { release!() }
}
```

This serializes all wallet write operations — auto-register and mint — so even if two requests arrive simultaneously, they execute sequentially. The second request waits for the first to finish (including transaction confirmation) before starting.

Second, `workers: 1` in the Playwright config. This eliminates all test-level concurrency. One test runs at a time. No worker A vs worker B race conditions.

The combination is insurance: the mutex protects against production concurrency (two real users minting at the same time), and single-worker tests eliminate false negatives from the test suite.

I also added friendly error message translation in the catch block. Instead of passing viem's raw error messages to the frontend, the MCP now maps them:

- `"replacement transaction underpriced"` → `"Already processing a mint — please wait a moment and try again."`
- `"SocketError"` → `"Connection interrupted — please try again."`
- `"execution reverted"` → `"Contract rejected the transaction (may already be minted)."`
- `"insufficient funds"` → `"MCP wallet gas funds depleted — contact admin."`

The frontend still gets the friendly message, and the raw error is logged server-side.

The test result after these changes: **25 passed, 2.2 minutes, zero retries.**

## The Singular Moment I Keep Coming Back To

It's not the OpenSea date fix or the BigInt error or the replacement-tx-underpriced saga. The moment I keep thinking about is the MCP deploy that failed because Railway's Nixpkgs removed Node 18.

The deploy log showed:

```
error: package 'nodejs-18_x' is not available in Nixpkgs
```

Node 18 reached end of life in April 2025. Railway's build system uses Nix under the hood, and they had pruned the EOL packages. The MCP was running on Node 18 and would never deploy again. The fix was a `.nvmrc` file with `22` in it, a `.nixpacks.toml` that pins the version, and a missing `bun.lock` that I had to generate by running `bun install` locally.

Fifteen minutes of work. But the experience of watching a build fail because a Node version hit end-of-life — something that happens on a fixed calendar schedule that I could have checked — was exactly the kind of "infrastructure rot" that the whole project is designed to outrun. The containers don't age. The token metadata is immutable. But the deployment pipeline rots in real time, and you either notice the Node 18 deprecation notification in your email or you don't.

I noticed because the deploy failed. That's not vigilance. That's luck.

## What I'd Tell The Person Who Wakes Up Tomorrow

Build the status endpoint last. Not because it's unimportant — it's the most-requested endpoint in the system — but because until you know what statuses you need, you'll build the wrong abstraction. I spent three versions of the status endpoint (on-chain iteration → Redis + on-chain hybrid → Redis-only) because I didn't know what question I was actually answering. The first version answered "is this container on chain?" The second answered "is this container in the Redis cache or on chain?" The third answers "is this container in the Redis cache?" and the answer is correct because Redis is updated atomically on every mint.

Also: fix the nonce management before you do anything else. A wrong nonce is the most expensive error because it doesn't fail at read time — it fails at write time, after you've already spent time and gas on the setup. Every other performance optimization is secondary to getting the nonce right.

## Key Takeaways

- **Nonces are the single source of truth, but they are also a single source of pain** — `getTransactionCount('pending')` before every write is slower than a nonceManager, but it never lies. The difference between "fast but occasionally wrong" and "correct always" is the line between a prototype and a production system.
- **Cache the lookup, not the result** — The Redis hash stores `containerId → tokenId`. The status endpoint just checks membership. This means the cache is populated once per token lifecycle (at mint time) and read many times. The data structure matches the access pattern, which is why it's fast.
- **Your RPC providers will betray you at 2 AM** — Use fallback transports with timeouts and `rank: false`. No single RPC provider is reliable enough for production on a mainnet chain. Test your fallback configuration by disconnecting the primary provider and confirming the system degrades gracefully.
- **A write mutex is cheap insurance** — A promise-based lock adds ~15 lines of code and eliminates an entire class of concurrency bugs. Every system that writes to a shared resource from concurrent handlers should have one.
- **The last 5% of reliability is 50% of the work** — Going from 90% pass rate (24/25 with retries) to 100% pass rate (25/25 with 0 retries) required: (1) a write mutex, (2) single-worker tests, (3) friendly error messages, (4) CoinGecko polling with stale data fallback, (5) Redis-only status endpoints, (6) fallback RPC providers. Each fix was small. The collective effect is a system that never makes you retry.
